# 埋点验证工具使用说明

## ⚠️ 重要：输入文件来源

| 规则 | 说明 |
|------|------|
| ✅ 必须使用 | 用户 @ 提供的 **埋点方案 xlsx** 和 **测试数据 csv** |
| ❌ 禁止使用 | `tracking-validator/source/` 下的内置示例文件 |
| 路径未给出 | 先向用户确认，**不得**自动 fallback 到 skill 内置文件 |

`source/` 目录中的文件仅供 Skill 开发时了解 Excel/CSV 格式，**不能用于正式验证**。

## 快速开始

### 1. 准备输入文件（用户提供）

- **埋点方案 Excel**：用户 @ 的文件，如 `【odirouter】埋点方案与开发计划.xlsx`
- **测试数据 CSV**：用户 @ 的文件，如 `odirouter_data.csv`（**仅需 5 列**）

#### 测试数据 CSV 必需列

| 列名 | 说明 |
|------|------|
| `st_event_name` | 事件名称 |
| `st_raw_message` | 完整上报 JSON（含 `properties`） |
| `st_status` | 入库状态（`1`=成功） |
| `st_error_info` | 失败原因（成功时可 `{}` 或留空） |
| `st_available_message` | 辅助 JSON（可为空，列须存在） |

表头示例：

```
st_event_name,st_raw_message,st_status,st_error_info,st_available_message
```

### 2. 指定序号

| 格式 | 示例 | 说明 |
|------|------|------|
| 单个 | `35` | 验证序号 35 |
| 范围 | `20-35` | 验证 20 到 35 |
| 列表 | `20,25,30` | 指定多个序号 |
| 混合 | `20-25,30,35` | 范围 + 列表 |
| 全量 | `1-175` | 全部点位 |

### 3. 执行验证

对 AI 说（需 @ 两个文件）：

```
@【odirouter】埋点方案与开发计划.xlsx @odirouter_data.csv 所有点位都验证下
```

命令行示例（**必须使用用户文件的绝对路径**）：

```bash
python skills/tracking-validator/scripts/validate.py \
  "/path/to/【odirouter】埋点方案与开发计划.xlsx" \
  "/path/to/odirouter_data.csv" \
  "1-175" \
  "/path/to/tracking_report_all.html"
```

执行时会打印：
```
=== 本次验证使用的文件（用户提供）===
埋点方案: /path/to/【odirouter】埋点方案与开发计划.xlsx
测试数据: /path/to/odirouter_data.csv
```

若误用 `source/` 内置文件，脚本会报错退出。

- **整表横向滚动**：列多时可左右拖动（`.table-wrap`）；表格 `width: max-content` + `<colgroup>` 固定列宽，避免列重叠

- **上报参数详情**：320px（埋点事件列 × 2.5），可自动换行
- **独立滚动区域**（固定高度，**横向 + 纵向**均可滚动，仅用于案例与错误/警告）：

## 判定逻辑

### 取最新一条与 case_record

| 用途 | 使用的记录 |
|------|-----------|
| 是否上报 / 是否入库 | 候选中 `st_event_time` **≤ 检验时刻** 且时间**最新**的一条（`latest`）；未来时间不参与 |
| 是否合规 / 入库案例 / 入库错误原因或警告 | 已入库时取 **最新 `st_status=1` 记录**（`case_record`，同样排除未来时间），基于其 `properties` **实时**校验 |
| 错误案例 | 未入库时取 `latest` 完整 JSON |

研发可能多次改代码上报，验证时按时间取最新记录作为判定依据；若最新事件时间落在未来（客户端时钟不准），则跳过，改用不超过检验时刻的最新一条。

### 入库 vs 合规 vs 警告

| 情况 | 是否入库 | 是否合规 | 报告展示 |
|------|----------|----------|----------|
| 最新一条 `st_status=1`，参数与嵌套结构完全符合方案 | ✅ | ✅ | 无警告/不合规 |
| 最新一条 `st_status=1`，嵌套键名错误（如 model_name 代替 model_id） | ✅ | ❌ | **不合规** |
| 最新一条 `st_status=1`，有多余顶层参数 | ✅ | ✅ | **警告**（仍合规） |
| 最新一条 `st_status≠1` | ❌ | ❌ | **错误原因** + 错误案例 |

### 错误原因 vs 不合规 vs 警告

**错误原因**（导致未入库）：
- `st_error_info` 系统校验错误
- 必填参数缺失

**不合规**（已入库但结构/参数不符）：
- 嵌套 JSON 键名与方案示例不一致
- 缺失**必填**参数（注释标注可选的不算）

```
不合规：【element_content】内应使用【model_id】，实际上报【model_name】
```

**警告**（不阻断入库，**不影响合规**）：
```
警告：参数【ref_page_id、current_page_id、status】上报参数详情中不要求上报
```

## 常见问题

### Q: 为什么不能用 source/ 下的示例文件？

示例文件是 Skill 开发时的格式参考，与用户当前方案和测试数据不同。正式验证必须使用用户 @ 提供的最新文件。

### Q: 为什么后端埋点（如 log_register_server）之前显示无数据？

Excel 中页面标识为空时，验证脚本**跳过** `current_page_name` 匹配，仅按事件名匹配。

### Q: 为什么 st_status=1 还有警告或不合规？

- **不合规**：嵌套结构键名与方案不一致（如 `element_content` 内应为 `model_id`，实际上报 `model_name`）
- **警告**：多余顶层参数仅作提醒，**不影响入库与合规判定**
- 建议更新 Excel 方案或修正上报字段

### Q: element_content 内 model_id 上报成 model_name 怎么判？

方案示例定义 `{model_id: "xxx"}`，实际上报 `[{"model_name":"xxx"}]`：
- **是否入库**：仍看 `st_status`（可能 ✅）
- **是否合规**：❌，归入「已上报已入库不合规」Tab

### Q: 五个 Tab 怎么划分？

- **已上报已入库合规**：最新一条 `st_status=1`，且无缺失必填参数、无嵌套键名错误（含仅有多余参数警告的点位）
- **已上报已入库不合规**：最新一条 `st_status=1`，但缺失必填参数或嵌套键名错误
- **已上报未入库**：有匹配记录，但最新一条 `st_status≠1`
- **未上报**：无匹配记录
- 恒等式：`总点位数 = 已上报已入库合规 + 已上报已入库不合规 + 已上报未入库 + 未上报`

### Q: 为什么入库案例里看不到某字段，但警告里却出现了？

**不应再出现此问题。** 入库案例与入库错误原因或警告共用同一条 `case_record`，基于同一 `properties` 实时计算。方案外多余参数会在入库案例「多余参数（方案未定义）」区块展示，与警告字段一一对应。

### Q: 为什么有数据但显示未入库？

看**错误原因**（不是警告/不合规）：通常是系统拒绝（属性不在方案中）或缺少必填参数。

### Q: 页面级旧点位与全局汇总点位（如 91 vs 120）怎么判？

各序号独立对照各自 Excel 白名单。版本迭代后实际上报多了新字段（如 `element_tab`），旧点位显示**警告**但仍**合规**；全局点位中注释为可选的参数（如 `model_id`）缺失不算不合规。

- `_client` 结尾 → 前端
- `_server` 结尾 → 后端

## 技术细节

### 匹配三要素

1. `st_event_name` = 埋点事件
2. `current_page_name` = 页面标识（**Excel 为空则跳过**）
3. 上报参数详情中的非动态固定值（如 `element_module=reward_task`）

### 预置属性和公共属性（不参与多余参数校验）

- **预置属性（硬编码）**：`st_role_id`、`st_account_id`、`st_distinct_id`、`st_event_name`、`st_event_time`
- **公共属性**：必须从 Excel「预置属性和公共属性」Sheet 的「属性名称」列读取（如 `platform_type`、`adid`、`utm_source`、`channel`）

### 必填 vs 可选参数

从「上报参数详情」每行解析参数名时，满足**任一**条件即为可选：

**方式一：注释标注**

| 注释短语 | 示例 |
|----------|------|
| 有就报，没有就算了 / 没有就算了 | `model_id={模型id} # 有就报，没有就算了` |
| 非必填 | `work_id={作品id} # 非必填` |
| 独立标注可选 / 选填 | `work_id={作品id} # 可选`（勿在描述里写「可选场景」「可选择」） |

**方式二：值侧花括号占位（有没有上报都无所谓）**

| 写法 | 说明 |
|------|------|
| `xxx = {可选参数}` | 花括号内语义为「可选」 |
| `xxx={可选}` | 简写 |
| `xxx={选填参数}` | 花括号内语义为「选填」 |
| `element_content={可选补充}` | 有补充内容时才报 |

> 普通动态占位（如 `ref_page_url={前页的url}`）不含「可选/选填」语义，仍为**必填**。

### 合规判定

| 检查项 | 影响合规 |
|--------|----------|
| 缺失**必填**参数 | ❌ 不合规 |
| 缺失**可选**参数 | 忽略 |
| 嵌套 JSON 键名错误 | ❌ 不合规 |
| 多余顶层参数 | 仅**警告**，仍合规 |

### 嵌套结构合规校验

1. 从「上报参数详情」示例块提取嵌套键名（如 `{model_id: "gpt-4o", pos: 0}`）
2. 将 JSON 字符串字段（如 `element_content`）解析后，检查数组内每个对象的键名
3. 方案要求 `model_id`，实际上报 `model_name` → **不合规**（序号46 典型场景）
