# 埋点点位上报入库验证

根据用户 **@ 提供的埋点方案 Excel** 和 **测试数据 CSV**，验证指定序号点位的上报与入库状态，输出 **HTML 验证报告**。

**验证范围：** Excel「开发计划」Sheet 中**所有点位**，不区分前后端——只要 Excel 里有的都要验。前端 `_client` 与后端 `_server` 事件使用同一套规则。

## ⚠️ 输入文件规则（强制，最高优先级）

1. **必须使用用户 @ 提供的 xlsx 和 csv**，使用其绝对路径执行验证
2. **禁止使用** `tracking-validator/source/` 下的任何文件作为验证输入，包括但不限于：
   - `source/副本-【odirouter】埋点方案与开发计划.xlsx`
   - `source/odirouter_test_data.csv`
3. 用户未 @ 文件或未给出路径时，**必须先询问**，不得 fallback 到 skill 内置文件
4. 执行前在终端打印本次使用的文件路径；报告页脚也会写入这两个路径
5. `validate.py` 会对 `skill/source/` 路径做拦截，误用会直接报错退出

**典型用户文件（示例，以用户实际 @ 为准）：**
```
/path/to/【odirouter】埋点方案与开发计划.xlsx
/path/to/odirouter_data.csv   # 仅需 5 列，见「测试数据 CSV 格式」
```

## 测试数据 CSV 格式（强制）

用户提供 CSV 时**只需以下 5 列**（表头名称须完全一致）：

| 列名 | 必填 | 说明 |
|------|------|------|
| `st_event_name` | ✅ | 事件名称，用于与 Excel「埋点事件」匹配 |
| `st_raw_message` | ✅ | 完整上报 JSON（含 `properties` 及业务参数） |
| `st_status` | ✅ | 入库状态（`1`=成功，其他=失败/未入库） |
| `st_error_info` | ✅ | 入库失败原因（JSON；成功时可留空或 `{}`） |
| `st_available_message` | ✅ | 可读事件时间等辅助信息（JSON；可为空，列须存在） |

**表头示例：**
```
st_event_name,st_raw_message,st_status,st_error_info,st_available_message
```

**`st_raw_message` 内须包含：**
- 顶层：`st_event_name`、`st_event_time`、`st_pk_id`、`st_user_id`、`st_role_id`、`st_account_id`、`st_distinct_id`
- `properties`：业务参数（如 `current_page_name`、`element_module` 等）

**`st_available_message` 取值优先级（用于报告 `st_event_datetime`）：**
1. JSON 内的 `st_event_datetime`
2. 若为空，由 `st_raw_message` 中 `st_event_time`（毫秒时间戳）转换

`validate.py` 启动时会校验 CSV 是否包含上述 5 列；含额外列时仅提示，不影响验证。

## 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| excel_file | ✅ | 用户 @ 提供的埋点方案 Excel **绝对路径** |
| csv_file | ✅ | 用户 @ 提供的测试数据 CSV **绝对路径**（**仅需 5 列**，见下方） |
| 序号范围 | ✅ | 要验证的序号，支持格式：`20-35`、`20,21,22`、`20-25,30,35`、`1-175`（全量） |
| output_file | ❌ | 输出 HTML 路径；默认建议放在用户数据同目录，如 `tracking_report_all.html` |

## 工作流程

```
确认用户文件路径 → 解析Excel → 提取埋点定义 → 解析CSV → 匹配数据 → 校验参数 → 生成报告
```

### Step 0：确认用户输入文件（必做）

1. 从用户消息中提取 @ 的 xlsx / csv 路径
2. 确认文件存在且**不在** `tracking-validator/source/` 下
3. 向用户明确列出本次使用的两个文件路径后再执行

### Step 1：解析埋点方案Excel

从Excel文件的「开发计划」Sheet提取以下字段：

| 列名 | 说明 |
|------|------|
| 序号 | 点位序号 |
| 埋点信息 | 点位中文名称 |
| 页面标识 | page_id |
| 埋点事件 | 事件名称（如 log_click_client） |
| 上报时机 | 触发时机描述 |
| 上报参数详情 | 定义的上报参数列表 |
| 前后端埋点 | 前端上报/后端上报 |

**参数解析规则：**
- 从「上报参数详情」列提取参数名，区分**必填**与**可选**
- **可选参数**识别（满足任一即可，缺失不算不合规）：
  - 行内 `#` 注释含「有就报，没有就算了」「没有就算了」「非必填」
  - 行内 `#` 注释为独立标注「可选」「选填」（如 `# 可选`），不误匹配「可选场景」「可选择」等描述
  - 值侧为花括号占位且语义为可选，如 `xxx = {可选参数}`、`xxx={可选}`、`xxx={选填参数}`、`element_content={可选补充}`
- 排除注释内容（`#` 后面的内容）用于取值解析
- 识别动态参数（如 `{点击的模型id}`）——仅当花括号内**不含**「可选/选填」时仍为必填
- 解析结果存入 `定义参数`（全部）、`必填参数`、`可选参数`

### Step 2：匹配测试数据CSV

在CSV文件中查找候选记录，再按「上报参数详情」做参数白名单校验。

**候选匹配：**
1. `st_event_name` = 埋点事件名称
2. `properties.current_page_name` = 页面标识（**页面标识为空时跳过**，适用于后端 `_server` 埋点）
3. 「上报参数详情」中定义的非动态固定值参数（如 `element_module=reward_task`、`element_pos=left_nav`）

**匹配条件解析规则：**
- 从Excel「上报参数详情」列直接解析参数名和值
- 跳过动态参数（值包含 `{}` 的参数）
- 跳过 `current_page_name`（已在步骤2中单独匹配）
- 跳过注释内容（`#` 后面的内容）

**CSV 必需列（仅此 5 列）：**
| 字段 | 说明 |
|------|------|
| st_event_name | 事件名称 |
| st_raw_message | 完整上报数据（JSON） |
| st_status | 状态（1=入库成功） |
| st_error_info | 错误信息（JSON） |
| st_available_message | 辅助信息（JSON，可为空；用于 `st_event_datetime`） |

### Step 3：校验参数并判定上报/入库/合规

**每个点位的参数白名单 = 该点位「上报参数详情」中解析出的参数名列表（必填 + 可选）。**

对比实际上报参数与方案定义，用于生成**错误原因**、**不合规**或**警告**：

> **合规判定原则：** 仅缺失**必填**参数或嵌套键名错误 → 不合规；多余参数 → 仅警告，不影响合规。

| 检查项 | 规则 | 归类 |
|--------|------|------|
| 多余参数 | 实际上报了白名单外的业务参数 | **警告**（不影响合规） |
| 缺失必填参数 | 白名单中**必填**参数未上报 | **错误原因**；同时导致**不合规** |
| 缺失可选参数 | 注释标注可选，或值侧为 `{可选参数}` 等占位 | 忽略 |
| 嵌套结构键名错误 | JSON 字符串字段（如 `element_content`）内对象键名与方案示例不一致 | **不合规** |
| 系统校验错误 | `st_error_info` 中的入库失败信息 | **错误原因** |
| 预置属性和公共属性 | 预置属性硬编码：`st_role_id`、`st_account_id`、`st_distinct_id`、`st_event_name`、`st_event_time`；其余公共属性**必须**从 Excel「预置属性和公共属性」Sheet 获取（如 `platform_type`、`adid`、`utm_source`、`channel`）；均不参与多余参数校验 | — |

**嵌套结构合规校验：**
- 从「上报参数详情」中的示例块提取嵌套键名（如 `{model_id: "xxx"}` → 期望键 `model_id`）
- 将实际上报的 JSON 字符串（如 `element_content`）解析后，检查数组内每个对象的键名
- 方案要求 `model_id`，实际上报 `model_name` → **不合规**（即使 `st_status=1` 已入库）

**典型场景（序号46 模型页-模型列表曝光）：**
- 方案 `element_content` 示例为 `[{model_id: "deepseek-v4-flash"}, ...]`
- 实际上报 `[{"model_name":"claude-haiku-4-5"}, ...]`
- `st_status=1` → **是否入库 ✅**；键名错误 → **是否合规 ❌**

**是否上报 / 是否入库 / 是否合规判定（取最新一条）：**
- 每个点位在全部候选记录中，按 `st_event_time` **取最新一条**判定是否上报/是否入库
- **排除未来时间**：`st_event_time` **晚于检验时刻**（脚本执行时的当前时间）的记录**不参与**「取最新」判定（客户端时钟不准或脏数据常见）；在「不超过检验时刻」的候选中再取最新
- 候选中若**仅有**未来时间记录：仍算**已上报**，但**入库=否**（无法作为有效判定样本），错误原因注明事件时间晚于检验时刻
- **入库案例**与**入库错误原因或警告**共用同一条 `case_record`（已入库时 = 最新 `st_status=1` 且事件时间≤检验时刻的记录），基于该记录 `properties` **实时**计算合规，保证两列一致
- **是否上报** ✅：存在匹配记录（含仅有未来时间记录的情况）
- **是否入库** ✅：判定用最新一条（已排除未来时间）`st_status=1`（多余参数仅警告，仍算入库成功）
- **是否入库** ❌：无匹配记录，或判定用最新一条 `st_status≠1`，或候选全为未来时间
- **是否合规** ✅/❌：无缺失必填参数、无嵌套键名错误即合规；多余参数仅警告

**典型场景（序号93 vs 94）：**
- 序号93 定义含固定值 `element_pos=left_nav` → 作为匹配条件，侧边栏点击匹配序号93
- 序号94 无 `element_pos` 匹配条件 → 同页同事件的侧边栏点击也可能匹配为候选；若 `st_status=1` 则入库成功，多余 `element_pos` 显示为**警告**（仍合规）

**典型场景（序号91 vs 120，版本迭代）：**
- 序号91（页面级旧规范）不含 `element_tab`；序号120（全局汇总新规范）含 `element_tab`，`model_id` 注释为「有就报，没有就算了」→ 可选
- 新版本上报多了 `element_tab` → 序号91：**合规 ✅ + 警告**；序号120：**合规 ✅**
- 无 `model_id` → 序号120/121 仍**合规 ✅**（可选参数缺失忽略）

**典型场景（序号1 log_register_server）：**
- Excel 页面标识为空 → 仅按事件名 `log_register_server` 匹配
- 实际上报含 `register_channel` 及多个方案未定义字段
- `st_status=1` → **是否入库 ✅**；多余字段 → **警告**（仍合规）

### Step 4：生成错误原因、不合规或警告

分多行展示，互不混用：

1. **错误原因**：`st_error_info` 系统校验错误；白名单中缺失的必填参数
2. **不合规**：嵌套 JSON 结构键名与方案示例不一致（如 `element_content` 内应为 `model_id`，实际上报 `model_name`）；缺失**必填**参数
3. **警告**：上报了白名单外、但已随数据入库的多余业务参数（**不影响合规**）

**格式示例（已入库但不合规）：**
```
不合规：【element_content】内应使用【model_id】，实际上报【model_name】
警告：参数【element_name、element_pos】上报参数详情中不要求上报
```

**格式示例（已入库且合规）：**
```
无
```

**格式示例（未入库）：**
```
错误原因：属性名【current_page_name】在埋点方案中不存在；必填参数【model_id】缺失
警告：参数【element_pos】上报参数详情中不要求上报
```

### Step 5：生成HTML报告

输出表格 HTML 报告，**整表支持横向滚动**，便于对照方案与案例。

**汇总栏（报告顶部）：**

| 指标 | 说明 |
|------|------|
| 总点位数 | 本次验证的点位总数 |
| 已上报已入库合规 | 上报 ✅、入库 ✅、合规 ✅ |
| 已上报已入库不合规 | 上报 ✅、入库 ✅、合规 ❌ |
| 已上报未入库 | 上报 ✅、入库 ❌ |
| 未上报 | 上报 ❌ |

恒等式：`总点位数 = 已上报已入库合规 + 已上报已入库不合规 + 已上报未入库 + 未上报`

**Tab 切换（汇总栏下方）：**

| Tab | 说明 |
|-----|------|
| 全部 (N) | 所有验证点位 |
| 已上报已入库合规 (N) | 上报 ✅、入库 ✅、合规 ✅ |
| 已上报已入库不合规 (N) | 上报 ✅、入库 ✅、合规 ❌ |
| 已上报未入库 (N) | 上报 ✅、入库 ❌ |
| 未上报 (N) | 上报 ❌ |

- `是否上报=✅` **且** `是否入库=✅` **且** `是否合规=✅` → 「已上报已入库合规」
- `是否上报=✅` **且** `是否入库=✅` **且** `是否合规=❌` → 「已上报已入库不合规」
- `是否上报=✅` **且** `是否入库=❌` → 「已上报未入库」
- `是否上报=❌` → 「未上报」
- 各 Tab 为空时显示对应提示文案（如「暂无未上报的点位」）

**表格列（按顺序）：**

| 列名 | 说明 |
|------|------|
| 序号 | 点位序号 |
| 埋点信息 | 点位中文名称 |
| 页面标识 | page_id（后端埋点为空显示 `—`） |
| 埋点事件 | 事件名称（`.col-event`，128px 宽，可换行） |
| 上报参数详情 | Excel 方案原文（`.col-param`，宽度为埋点事件列 × 2.5，可换行） |
| 是否上报 | ✅ 是 / ❌ 否 |
| 是否入库 | ✅ 是 / ❌ 否 |
| 是否合规 | ✅ 是 / ❌ 否 |
| 入库案例 | `case_record` 的 JSON（510px 宽，独立滚动）；已入库时 = 最新 `st_status=1` 记录 |
| 入库错误原因或警告 | 基于**同一条** `case_record` 实时校验；已入库时仅展示不合规/警告 |
| 错误案例 | 最新一条未入库记录的 JSON；已入库为「无」（510px 宽） |

**CSS 列宽变量（`table` 上定义）：**

| 变量 | 值 | 适用列 |
|------|-----|--------|
| `--col-event-width` | 128px | 埋点事件 |
| （计算） | 320px（×2.5） | 上报参数详情 |
| `--col-case-width` | 510px（原 340px × 1.5） | 入库案例、错误案例、入库错误原因或警告（三列同宽） |

**报告布局规则：**
- 表格外层 `.table-wrap` 包裹，列宽超出时可**横向拖动**浏览
- 表格使用 `width: max-content` + `<colgroup>` 固定列宽，避免 `width: 100%` 导致列重叠
- 「埋点信息」列（`.col-name`）允许**自动换行**，避免单行过长撑开表格
- 「埋点事件」列（`.col-event`）：固定宽度 `--col-event-width`（128px），**可自动换行**
- 「上报参数详情」列（`.col-param`）：宽度 `calc(--col-event-width × 2.5)`（320px），**可自动换行**（`pre-wrap`），不设滚动条
- 「入库案例」「错误案例」「入库错误原因或警告」使用 **`.scroll-box` 独立滚动区域**（固定高度，横纵双向滚动）：

| 区域 | CSS 类 | 高度 |
|------|--------|------|
| 入库案例 / 错误案例 | `.scroll-box.case-box` | 180px |
| 入库错误原因或警告 | `.scroll-box.error-box` | 120px |

列宽：`--col-case-width: 510px`（原 340px × 1.5）；入库案例、错误案例、入库错误原因或警告三列同宽。

- 「入库案例」「错误案例」内 `pre.case-json`：
  - 每个 `"参数": 值` **独占一行**
  - 行内**不自动换行**（`white-space: pre`）
  - 长内容由外层 `.scroll-box` **横向/纵向滚动**查看
- 渲染由 `render_case_box()` / `render_error_box()` 生成滚动容器

**列样式速查：**

| 列 | CSS 类 | 展示方式 |
|----|--------|----------|
| 埋点信息 | `.col-name` | 自动换行 |
| 埋点事件 | `.col-event` | 128px 宽，可换行 |
| 上报参数详情 | `.col-param` + `pre.spec-detail` | 320px（×2.5），可换行 |
| 入库案例 / 错误案例 | `.col-case` + `.scroll-box.case-box` | 510px 宽，180px 高 |
| 入库错误原因或警告 | `.col-error` + `.scroll-box.error-box` | 510px 宽，120px 高 |

**动态参数采样（汇总型点位）：**
- 对于含动态参数（值侧为 `{...}` 占位的参数，如 `current_page_name = {页面标记}`）的汇总型点位
- 从所有候选记录中收集每个动态参数的实际取值，去重后最多展示 20 条
- 展示在入库案例的 JSON 下方，格式为：

## 常见问题

### Q0: 能否使用 skill/source/ 下的示例文件验证？
**不能。** 必须使用用户 @ 提供的最新方案和测试数据。`source/` 仅作格式参考。

### Q1: 为什么某个序号没有匹配到数据？
检查：
1. 埋点事件名称是否正确
2. 页面标识（current_page_name）是否匹配（后端埋点页面标识为空则跳过此项）
3. 是否有固定值匹配条件（如 `element_module=reward_task`）

### Q2: 为什么有数据且 st_status=1 但报告里显示不合规或有警告？
- **不合规**：嵌套结构键名与方案不一致（如 `element_content` 内应为 `model_id`，实际上报 `model_name`），或缺失必填参数
- **警告**：方案未定义的多余顶层参数（如 `element_pos`），不影响入库与合规判定
- 建议修正上报字段或更新 Excel 方案

### Q3: 为什么有候选数据但显示未入库？
检查**错误原因**（非警告）：
1. 系统校验错误（`st_error_info`，如属性名在方案中不存在）
2. 缺少方案要求的必填参数（`st_status≠1`）

### Q4: 如何区分前端和后端上报？
- 事件名以 `_client` 结尾 → 前端上报
- 事件名以 `_server` 结尾 → 后端上报
- **验证时不区分**：Excel 里有的前后端点位都要验，后端埋点页面标识通常为空，仅按事件名匹配

### Q5: 五个 Tab 怎么划分？
- **已上报已入库合规**：判定用最新一条（事件时间≤检验时刻）`st_status=1`，且无缺失必填参数、无嵌套键名错误（含仅有多余参数警告的点位）
- **已上报已入库不合规**：判定用最新一条 `st_status=1`，但缺失必填参数或嵌套键名错误
- **已上报未入库**：存在匹配记录，但判定用最新一条 `st_status≠1`，或候选全为未来时间
- **未上报**：测试数据中无匹配记录
- 恒等式：`总点位数 = 已上报已入库合规 + 已上报已入库不合规 + 已上报未入库 + 未上报`

### Q6: 为什么有一条「更新」的失败记录，历史却大量入库成功，报告仍显示未入库？
旧逻辑按绝对最新 `st_event_time` 判定；若最新一条是**未来时间**（客户端时钟不准），常因 24h 时效校验失败，会误判整点位未入库。
**现行规则**：`st_event_time` 晚于检验时刻的记录不参与「取最新」；在不超过检验时刻的候选中再取最新。未来时间失败记录会被排除，历史成功入库可正常判定为已入库。

### Q7: element_content 内 model_id 上报成 model_name 怎么判？
方案示例中定义了 `{model_id: "xxx"}`，实际上报 `[{"model_name":"xxx"}]` → **是否入库** 仍看 `st_status`；**是否合规** ❌，归入「已上报已入库不合规」Tab，并显示：
```
不合规：【element_content】内应使用【model_id】，实际上报【model_name】
```

### Q8: 为什么入库案例里看不到某字段，但警告里却出现了？
**不应再出现此问题。** 入库案例与入库错误原因或警告共用同一条 `case_record`，基于同一 `properties` 实时校验。方案外多余参数会在入库案例的「多余参数（方案未定义）」区块展示，与警告一一对应。

### Q9: 页面级旧点位（如91）与全局汇总点位（如120）怎么判？
各序号**独立**对照各自 Excel 白名单：
- 旧点位多了新字段（如 `element_tab`）→ **警告**，仍合规（版本迭代常见）
- 全局点位含可选参数（如 `model_id # 有就报，没有就算了`）→ 缺失不算不合规
- 建议：版本迭代后同步更新页面级点位的 Excel 定义，或依赖全局汇总点位（120/121）做新规范验收
